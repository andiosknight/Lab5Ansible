# Отчет по лабораторной работе №6.2: Управление конфигурацией сервера с помощью Ansible

## Цель работы
Получение практических навыков автоматизации развертывания и управления конфигурацией серверов Linux с использованием Ansible, проектирование кастомных ролей и применение методологии IaC (Infrastructure as Code) для деплоя веб-приложений.

---

## Ход выполнения работы

### Шаг 1. Проектирование структуры и настройка локального инвентаря

1. На сервере Ubuntu создана классическая структура директорий Ansible проекта, включающая главный сценарий, файл инвентаря и папку `roles` с изолированными подкаталогами для каждой роли:
   * `base` — базовая подготовка системы.
   * `nginx` — развертывание веб-сервера.
   * `app` — автоматический деплой статического сайта.
   * `ssh` — импорт публичных ключей доверенных хостов.
2. Так как запуск сценария автоматизации выполняется непосредственно внутри целевой системы, настроен файл инвентаризации `inventory.ini` со следующим содержимым:
   ```ini
   [webservers]
   localhost ansible_connection=local
   ```

### Шаг 2. Разработка Ansible-плейбука и конфигурация ролей

1. **Главный файл сценария `setup.yml`**: Связывает группу хостов `webservers` со всеми ролями и назначает индивидуальные теги (`tags: ["app"]` и т.д.) для обеспечения возможности раздельного запуска.
2. **Роль `base`**: Описаны задачи по установке утилит (`curl`, `git`, `unzip`) и защитного ПО `fail2ban`
3. **Роль `nginx`**: Реализован сценарий установки веб-сервера Nginx и добавления его в автозагрузку операционной системы.
4. **Роль `app`**: Сначала модуль `file` полностью зачищает старую директорию и выставляет корректные права владельца (`www-data`), а затем модуль `copy` генерирует чистый структурированный файл `index.html` со встроенными стилями прямо в корень веб-сервера.
5. **Роль `ssh`**: С помощью модуля `authorized_key` настроен автоматический импорт сгенерированного на хосте публичного ключа (алгоритм `ed25519`) в профиль суперпользователя на сервере.

### Шаг 3. Запуск и отладка пайплайна

   ```bash
   ansible-playbook -i inventory.ini setup.yml -K
   ```  

### Шаг 4. Тестирование изолированного деплоя по тегам (Tags)

1. Для проверки корректности работы механизма тегов была запущена команда обновления исключительно веб-приложения:
   ```bash
   ansible-playbook -i inventory.ini setup.yml -K --tags "app"
   ```
2. Ansible успешно проигнорировал системные роли `base`, `nginx` и `ssh`, выполнив только две целевые задачи: `Очистка старой директории Nginx перед деплоем` и `Создание статической страницы index.html`.
3. Сценарий завершился со статусом `failed=0`, подтверждая стопроцентную работоспособность логики тегов.

### Шаг 5. Верификация результатов автоматизации

1. Проверен статус развернутых служб на сервере — службы `nginx` и `fail2ban` успешно запущены и функционируют в фоновом режиме.
2. На хост-машине (Windows) открыт браузер и осуществлен переход по адресу `http://localhost:8080`.
3. Страница корректно отображает статический сайт, который Ansible автоматически импортировал из репозитория.

## Вывод
В ходе выполнения лабораторной работы были успешно освоены концепции Infrastructure as Code (IaC) с использованием систем управления конфигурациями на примере Ansible. Создание декларативных ролей позволило полностью абстрагироваться от ручного ввода императивных команд в терминале. Использование модулей `git` и `service` совместно с механизмом тегов продемонстрировало высокую гибкость, скорость деплоя и идемпотентность (повторяемость) автоматизированных сценариев администрирования.
